# DECISIONS — YMU-A

Decisions made during planning and Phase 0, with rationale. Product-level decisions (Zoho, calendar auth, GPS policy, offline trust, SMS, event↔school matching) were confirmed with the user before implementation and are recorded in the approved plan (`/Users/pepskq/.claude/plans/in-the-file-directory-cozy-sparrow.md`) — summarized at the bottom here for convenience. ## Implementation-level decisions (made during Phase 0 build)

### `@serwist/turbopack` instead of `@serwist/next`
The plan specified `serwist` generically. `create-next-app` on this Next.js version (16.2.10) produces a Turbopack-only project — there is no webpack build path to fall back to. `@serwist/next` requires webpack. Used **`@serwist/turbopack`** instead (same Serwist 9.5.11 release line, experimental Turbopack integration per Serwist's own docs). This changes the setup shape from the standard Serwist guide:
- Service worker source: `src/app/sw.ts` (imports `defaultCache` from `@serwist/turbopack/worker`, not `@serwist/next/worker`)
- Requires a dedicated route handler at `src/app/serwist/[path]/route.ts` using `createSerwistRoute` — this doesn't exist in the webpack setup
- `SerwistProvider` (from `@serwist/turbopack/react`) wraps the root layout to register the SW client-side, pointed at `/serwist/sw.js`
- Added `esbuild` as a dev dependency (required by `createSerwistRoute`'s `useNativeEsbuild: true` option)

Verified working: service worker registers and activates at root scope, manifest and offline fallback both precache correctly, confirmed via direct browser inspection (`navigator.serviceWorker.getRegistrations()`, Cache Storage contents) rather than just a successful build.

### Precache revision tied to git HEAD
`src/app/serwist/[path]/route.ts` shells out to `git rev-parse HEAD` for the offline-page precache revision, falling back to `crypto.randomUUID()` if git isn't available (e.g. a tarball deploy with no `.git`). This ties cache invalidation to actual deploys rather than a hardcoded string that could go stale.

### Placeholder icons generated with zero image dependencies
`scripts/generate-icons.mjs` hand-writes raw PNG bytes (manual IDAT/IHDR chunks + zlib deflate) rather than pulling in `sharp` or `canvas`. This was purely to avoid adding a heavy native-binary dependency for what is explicitly throwaway placeholder art (solid indigo square, white circle) — real branding will replace this before launch, at which point this script and its output should probably be deleted rather than maintained.

### Role-accent CSS variables added early, unused
`globals.css` defines `--accent` overrides keyed on `data-role="regional_manager|operations_manager|cpo"` even though nothing sets that attribute yet (no auth exists). This anticipates the plan's "different main color per role" requirement (PRD, Settings section) so Phase 1 only needs to set the attribute on `<html>`, not invent the color scheme.

### `.env.example` pre-populated with every phase's future variables
Rather than adding env var placeholders phase-by-phase, all of them (Google service account, VAPID, Resend, Zoho) were added to `.env.example` upfront with comments noting which phase introduces each, so the file only needs value backfilling later, not restructuring.

### CI builds with placeholder Supabase env vars
`.github/workflows/ci.yml` passes fake `NEXT_PUBLIC_SUPABASE_URL`/`ANON_KEY` values to `npm run build`. This is required because Next.js needs these to be defined at build time (they're inlined into client bundles) even though no real Supabase calls happen during a static build — without this, CI would fail on missing env vars unrelated to actual code correctness.

## Implementation-level decisions (made during Phase 1 build)

### `src/proxy.ts`, not `middleware.ts`
Next.js 16 renamed the middleware file convention to **proxy** (`node_modules/next/dist/docs/01-app/01-getting-started/16-proxy.md`; the old name is deprecated). The planned `middleware.ts` is therefore `src/proxy.ts` with an exported `proxy()` function — functionality identical. Per the Next auth guide, it does **optimistic checks only** (cookie/JWT, no DB queries); authoritative checks live in `src/lib/auth/dal.ts` and in RLS.

### Role rides in the JWT via `app_metadata.app_role`
The proxy needs the role without a DB round-trip on every navigation. A DB trigger stamps `app_role: 'teacher'` into `auth.users.raw_app_meta_data` at signup and `promote_user()` keeps it in sync. `app_metadata` is not client-writable. Consequence: after a promotion, the target's JWT is stale for up to one token refresh (~1 h) — acceptable because the proxy check is only optimistic; the DAL and RLS see the new role immediately.

### `data-role` on the (app) layout wrapper, not `<html>`
Phase 0 anticipated setting `data-role` on `<html>` from the root layout. That breaks in practice: the root layout does **not** re-render on soft navigations (the post-login server-action redirect is one), so the attribute would be stale until a hard reload. Instead the `(app)` route-group layout — which mounts fresh whenever navigation crosses from the (auth) group — sets `data-role` on its wrapper div, and the `globals.css` selectors were relaxed from `:root[data-role=…]` to `[data-role=…]`.

### Archived-account gate via `/auth/signout` route handler
Server components can't write cookies mid-render, so the DAL can't call `signOut()` directly when it finds `archived_at` set (the session would survive and redirect-loop). Archived users are redirected to `GET /auth/signout?error=archived`, a route handler that clears the session and lands on `/login` with the archived message. The login action additionally checks `archived_at` immediately after password verification, so an archived user can't even establish a session.

### RLS enabled but not FORCEd; column protection via trigger
`FORCE ROW LEVEL SECURITY` would make the `current_app_role()`/`current_app_region()` security-definer helpers recurse into the very policies that call them (table owner stops bypassing RLS). So: RLS is enabled, owner/service-role bypass is relied on for the helpers, table grants are cut to `select, update` for `authenticated`. Because RLS can't restrict individual columns, a `BEFORE UPDATE` trigger rejects changes to `role`/`region`/`archived_at` by non-OM/CPO callers (and deliberately lets JWT-less service-role sessions through).

### Promotion matrix enforced in the `promote_user()` RPC, mirrored in UI
OM may set teacher↔regional_manager (region required for RM, cleared otherwise); only CPO may appoint/demote operations managers; `cpo` is never assignable via RPC (manual seed only, `0003_seed_cpo.sql`); a CPO's role is immutable via RPC. The first guards shipped in `0002`, the target-role guards in `0004_promote_target_guards.sql` (gap noticed while building the UI: an OM could have demoted the CPO). The Team page mirrors these rules so the UI never offers a submit the RPC would reject.

### First OM is promoted by the CPO in-app (user-confirmed)
The Team page is available to **both** OM and CPO, so the manually-seeded CPO appoints the first Operations Manager through the UI. No OM seed migration exists.

### Built-in Supabase email sender for now (user-confirmed)
Signup verification uses Supabase's default sender — rate-limited (~2/hour) and it rejects throwaway domains like `example.com`. Fine for dev; the Resend SMTP cutover (a dashboard configuration, no code change) is documented in HANDOFF.md and must happen before onboarding real teachers.

### Never name a migration `*_init.sql`
`supabase db push` silently **skips** migration files whose name part is `init` ("replace \"init\" with a different file name to apply this migration"). Phase 0's `00000000000001_init.sql` was skipped this way; the enums turned out to already exist on the hosted project, so `0002` applied cleanly. Fixed by `supabase migration repair --status applied 00000000000001` and renaming the file to `00000000000001_base_enums.sql` (same version key, so nothing re-runs).

### RLS tests run against the hosted project, not CI
No Docker on this machine ⇒ no local Supabase. `tests/rls.test.ts` (vitest) creates disposable confirmed users on the **hosted** project via the service-role key and deletes them in `afterAll`. Kept out of CI: it needs live secrets and mutates the shared project's user list. Run manually with `npm run test:rls` after any RLS-touching migration.

## Implementation-level decisions (made during Phase 2 build)

### Region write-access is trigger-enforced, reusing the Phase 1 pattern exactly
"Region assignment OM/CPO only; immutable for RMs once set" is implemented identically to how Phase 1 locked `profiles.role`/`region`/`archived_at`: RLS lets any manager `UPDATE` a `schools` row (contact info, geofence radius, lat/lng override), and a `BEFORE UPDATE` trigger (`protect_school_region()`) specifically rejects a changed `region` column unless the caller is OM/CPO. The `INSERT` policy additionally requires `region is null` for non-OM/CPO callers, so an RM can add a school but never create one pre-assigned to a region. This means "immutable for RMs" is really "RMs can never write this column, full stop" — simpler than trying to distinguish "set" vs "changed" and matches the Phase 1 precedent closely enough that the RLS tests read almost like siblings of `tests/rls.test.ts`.

### `teacher_directory()` RPC instead of syncing email onto `profiles` (user-confirmed)
Profile popovers need a teacher's email, which lives in `auth.users`. Phase 1 punted this. Chose a `SECURITY DEFINER` SQL function over a sync trigger: no duplicated, driftable copy of auth data, and it follows the same helper pattern as `current_app_role()`/`current_app_region()`. Because `SECURITY DEFINER` bypasses RLS, the function manually re-implements the `profiles_select` region-scoping (RM: own-region teachers only; OM/CPO: everyone) rather than relying on the caller's RLS context.

### No teacher↔school assignment table in Phase 2 (user-confirmed)
The brief asked for "a teacher list per school," but no data links a teacher to a specific school yet — that's meant to come from Google Calendar event matching in Phase 3 (`calendar_events.teacher_id`/`school_id`). Rather than invent a manual assignment table that Phase 3 would then have to reconcile with calendar-derived links, Phase 2's Teachers list is grouped/filterable by region only (`profiles.region`, which already exists). See NEXT_STEPS.md for what Phase 3 should do once the real link exists.

### Leaflet marker icons served from `public/leaflet/`, not bundled via `import`
`react-leaflet`'s default marker icon references relative URLs that resolve against `leaflet`'s own package location, which 404s once bundled. The standard fix (`import markerIcon from "leaflet/dist/images/marker-icon.png"` then `L.icon({ iconUrl: markerIcon.src, ... })`) does not work on this Next.js/Turbopack version — it throws `iconUrl not set in Icon options` at runtime, discovered by actually opening `/lists` in a browser rather than trusting the build to catch it. Fixed by copying the three marker PNGs into `public/leaflet/` and referencing them as plain string paths (`/leaflet/marker-icon.png`). This also means the icons work offline once the PWA's service worker caches `public/`, which a CDN-hosted icon wouldn't.

### Migration numbered `0005`, not `0003` as the original brief said
The Phase 2 task brief (written before Phase 1 finished) said `supabase/migrations/0003_schools.sql`. By the time Phase 2 started, `0003` (CPO seed) and `0004` (promote target guards) were already taken — confirmed via `supabase migration list` and `NEXT_STEPS.md`, which was more current than the brief. Used `0005_schools.sql`.

### `school_years` created standalone, not yet wired to `schools`
The PRD schema line for `schools` mentions "school_year links," but nothing before Phase 9 (school-year lifecycle: create, link schedules/attendance, archive) actually consumes that relationship. Added `school_years` as its own table (name, start_date, end_date, archived) with RLS (OM/CPO write, all managers read) and left it unlinked rather than fabricating a foreign key nothing uses yet.

### Nominatim throttled with a module-level timestamp, not a queue
Nominatim's usage policy caps unauthenticated use at 1 req/s and requires a descriptive `User-Agent`. Since schools are added one at a time by a human (≤255 total, drip-fed per the plan's own risk register), a single `lastNominatimCallAt` timestamp with a `setTimeout` delay in `lib/geocode.ts` is sufficient — no request queue needed.

## Implementation-level decisions (made during Phase 3 build)

### Dependency-free, isomorphic Google client instead of `googleapis`
`src/lib/google/calendar.ts` signs the service-account JWT with WebCrypto (`crypto.subtle`, RS256) and calls the Calendar v3 REST API with `fetch` — no `googleapis` package. Reason: the identical code must run in Next.js (Node) **and** the Supabase Edge Function (Deno), and `googleapis` is Node-oriented and heavy. This matches the project's established dependency-minimalism (cf. the hand-written PNG encoder in Phase 0). The module is also written in **erasable-only TS syntax** (explicit class fields, not constructor parameter properties) so Node's native TS stripping executes it directly — that's what lets `scripts/sync-calendar.ts` import it with no build step. `tsconfig` gained `allowImportingTsExtensions` so the runner can import `sync.ts`/`calendar.ts` with explicit `.ts` extensions (required by Node's resolver); harmless for the app, which imports via the `@/` alias.

### Edge Function is the artifact; a Node runner shares the exact core (no Docker)
The sync logic lives in `supabase/functions/calendar-sync/sync.ts` as `syncCalendar(supabase, env)`, written isomorphic (clients passed in, not constructed). `index.ts` is the Deno `Deno.serve` wrapper for pg_cron. Because this machine has no Docker (so no `supabase functions serve`), `scripts/sync-calendar.ts` (`npm run sync:calendar`) drives the **same** `syncCalendar()` from Node against the hosted DB — that's the local verification path and the command for the end-to-end acceptance test. The Edge Function imports the shared client via a relative out-of-tree path (`../../../src/lib/google/calendar.ts`); the `supabase` CLI bundler follows it. `supabase/functions` is excluded from Next's `tsconfig`/ESLint (Deno modules, `npm:` imports), and `sync.ts`/`index.ts` carry `// @ts-nocheck`.

### syncToken handling: full → incremental, 410 → full resync, first-import notification suppression
No stored token ⇒ full sync (user chose to ingest **everything**, so no `timeMin`). A stored token ⇒ incremental. A `410 GONE` clears the token and re-runs a full sync **while preserving `full_synced_at`**. Change/removal notifications are gated on `Boolean(full_synced_at)` (`detectChanges`): the very first import establishes the baseline silently (no notifying every teacher about every existing class), but a post-410 recovery — which has a `full_synced_at` — still emits real changes. Full-sync **deletion reconciliation** finds events not re-touched this pass via `synced_at < syncStartedAt` (every seen event is stamped with the run's start time), marks them cancelled, and notifies — this is the only way a full sync learns about removals, since incremental sync gets explicit `cancelled` stubs.

### Fuzzy school match: pg_trgm, name-word-similarity ∨ address-similarity, threshold 0.5
`match_school()` normalizes both strings (lowercase, strip punctuation, collapse whitespace) and scores each school as `greatest(word_similarity(name, location), similarity(address, location))`. `word_similarity` (not plain `similarity`) on the name is deliberate: Google Location is usually `"School Name, 450 Bird Rd, City"`, i.e. the name is a substring of a longer string, which plain trigram similarity scores poorly. The 0.5 threshold lives in `sync.ts` (`SCHOOL_MATCH_THRESHOLD`), so it's tunable in one place; below it the event lands in the manager unmatched queue. A **manual** assignment (`school_match_source='manual'`) survives subsequent syncs unless the event's Location text itself changes, which re-runs the fuzzy match.

### `teacher_ids uuid[]`, region derived from the event's school (both user-confirmed)
An event's matched teachers are stored as an array of every attendee whose email matches a login email — the regular teacher and any substitute are simply two matched teachers; Google has no primary/substitute distinction to import, and a "teacher or substitute changed" notification falls out of an array set-difference. Teachers can span multiple regions, so **a teacher's region is derived from the schools their events are at**, not from `profiles.region` (left untouched). Consequently event RLS/visibility and the Schedules "by region" filter key off the event's `school.region`, and a `teacher_has_scheduled_school()` helper extends `schools_select` so a teacher can read the schools they're scheduled at (their own schedule + Phase 4 clock-in map).

### `notification_queue` generic + service-role-only; Phase 3 enqueues only
`type` + `payload` (jsonb) + `send_at` + `status`, so Phase 7's reminder types reuse the same table without a schema change. Phase 3 writes `time_changed`/`location_changed`/`teacher_changed`/`event_cancelled` rows; sending is Phase 7. The table has no authenticated grants at all (service-role writes from the sync); the RLS test asserts authenticated reads are denied.

### `assign_event_school` is `SECURITY DEFINER`; RMs confined to their region on both ends
Authenticated users have no UPDATE grant on `calendar_events`, so manual matching goes through this RPC and its role/region checks are the whole authorization story (same pattern as `promote_user`). A Regional Manager can only touch an event whose **current** school is in their region (or unassigned) and can only assign a **destination** school in their region; OM/CPO are unrestricted.

### Event detail renders the description as plain text, not Google's HTML
Google Calendar descriptions can contain HTML. The detail view renders `description` as `whitespace-pre-wrap` plain text rather than injecting HTML — no sanitizer dependency, no XSS surface. Trade-off: a description authored with rich HTML shows its tags literally; acceptable for a scheduling tool and reversible later if a vetted sanitizer is added.

### Migration numbered `0006`, not `0004` as the brief said
The Phase 3 brief (written before Phase 1 finished) said `0004_events.sql`; `0004` was taken by `promote_target_guards` and `0005` by schools. Used `0006_events.sql`, consistent with `NEXT_STEPS.md`.

## Implementation-level decisions (multi-calendar sync, built after Phase 3)

Reused what worked from a companion repo (`School-Visit-Planning-YMU`, which solved a version of this problem already) and rebuilt what didn't: its schema shape (a pinned `googleCalendarId` on the school row, an unmatched-calendar review table) and its syncToken/410-handling code were worth keeping; its matching algorithm (naive exact-string match, no admin UI actually wired up, `calendarList.list()` with no pagination loop) was not — this extends YMU-A's own pg_trgm fuzzy match and manual-override RPC/UI pattern (already proven at the event level) to the calendar level instead, one layer above the existing event↔school match, which is unchanged.

### Pin-then-skip, not periodic re-validation (user-confirmed)
Once a calendar is matched to a school (`schools.google_calendar_id` set), later syncs never re-match it — a manager must explicitly relink it via `resolve_calendar_issue()`. The alternative (re-run the fuzzy match against every already-pinned school on every tick, flagging a `match_degraded` issue on drift) was presented and explicitly declined in favor of simplicity, matching the reference repo's approach. Risk accepted: a renamed calendar or a bad manual link won't self-flag; only a fresh calendar discovery can raise a new issue.

### Calendar-match threshold is a separate, untuned constant (user-confirmed)
`CALENDAR_MATCH_THRESHOLD` (0.5) and `AMBIGUITY_MARGIN` (0.08) in `sync.ts` are distinct from `SCHOOL_MATCH_THRESHOLD` — that constant was tuned for free-text event `Location` strings, while a calendar `summary` is a short, structured name with different score characteristics. Both are placeholders and **must be validated against the real planned ~30-70 calendar-summary strings** before the sync is relied on in production (see `classifyDiscoveredCalendar` in `sync.ts`, unit-tested in `tests/calendar-sync-classify.test.ts` specifically so this validation can happen offline with synthetic data, before any calendar exists).

### Column-level `REVOKE`, not a trigger, protects the new `schools` columns
`protect_school_region()` (0005) blocks a column via a `BEFORE UPDATE` trigger checking `auth.uid() is not null and role not in (om, cpo)` — that works because region is only ever written through an ordinary RLS-gated client `UPDATE`, never through an intermediate RPC. The four new calendar-match columns are different: the only intended write path besides the service-role sync is `resolve_calendar_issue()`, a `SECURITY DEFINER` RPC called *by* an authenticated manager — inside it, `auth.uid()` is **not** null (it's the calling manager's id; `SECURITY DEFINER` changes the effective role for privilege checks, not the JWT-derived session GUCs `auth.uid()` reads from). A trigger keyed on "`auth.uid()` is not null" would therefore block the RPC's own legitimate writes, not just a raw client bypass — caught before shipping by tracing through what `auth.uid()` actually resolves to inside a `SECURITY DEFINER` call, not just by pattern-matching the existing trigger. Fixed with `REVOKE UPDATE (col, ...) / REVOKE INSERT (col, ...) ... FROM authenticated` instead: column-level privilege checks run against the *executing* role (`authenticated` for a raw client call vs. the function owner for the RPC/migrations), never against `auth.uid()`, so they block direct writes while leaving the RPC (and the service-role sync, which already has a blanket `grant all` on `schools`) unaffected.

### `calendar_sync_issues.calendar_id` is a plain unique column, not a partial "open issues only" index
The natural design — a partial unique index `on (calendar_id) where resolved_at is null`, so a resolved issue doesn't block a fresh row on rediscovery while still deduplicating *open* issues — doesn't compose with Supabase/PostgREST's upsert: `.upsert(row, { onConflict: "calendar_id" })` generates a plain `ON CONFLICT (calendar_id) DO UPDATE`, which cannot target a partial index (Postgres requires the `ON CONFLICT` clause's own predicate to match, and PostgREST's upsert helper has no way to express one). Used a plain `unique` constraint on `calendar_id` instead: one row per calendar ever, and rediscovering a previously-resolved calendar as an issue again simply reopens that same row (`resolved_at` cleared) rather than inserting a duplicate — functionally equivalent for this table's purpose, and it upserts correctly.

### Single-row lease table instead of `pg_advisory_lock` for overlap prevention
`calendar_sync_lock` is a one-row table (`id boolean primary key`, `locked_until`) acquired via an atomic `UPDATE ... WHERE locked_until IS NULL OR locked_until < now()`. Not `pg_advisory_lock`: Supabase Edge Function DB connections go through a pooler and aren't guaranteed to hold one long-lived session for the life of a request, which makes session-scoped advisory locks unreliable there. The lease row works over stateless HTTP calls and self-heals (the lease simply expires) if a run crashes without releasing it.

### `calendarList` vs ACL: sharing a calendar with the service account does not make it discoverable (found live, against the real service account and real school calendars)
Discovered during first-run validation: after bulk-sharing ~68 real school calendars with the service account (Apps Script, ACL grants confirmed working — direct `events.list` against a shared calendar succeeded), `syncAllCalendars`'s discovery still reported `discovered: 0`. Root cause: Google Calendar's ACL (who can access a calendar) and a user/service-account's **calendarList** (their own personal "list of calendars I follow") are two separate resources. Sharing a calendar with a human auto-subscribes it to their calendarList because the Calendar UI does that on accept; a service account has no UI to "accept" anything, so an ACL grant alone never populates its calendarList, and `calendarList.list()` (what `listAllCalendars()` in `calendar.ts` uses for discovery) stays empty regardless of how many calendars are actually shared.
- **Fix**: added `GoogleCalendarClient.subscribeToCalendar(calendarId)` (`POST /users/me/calendarList`, 409-tolerant/idempotent) and a reusable bootstrap script, `scripts/subscribe-calendars.ts <calendar-ids.json>`, that explicitly subscribes the service account to a known list of calendar IDs. This must be run once after any batch of new calendars is shared (documented in `NEXT_STEPS.md` as part of the onboarding-a-school runbook) — there is no Calendar API endpoint that lets a service account enumerate "every calendar I have ACL access to" independent of calendarList, so a human-provided ID list (from the Apps Script's own log output, which already prints each `name (calendarId)` pair) is the only source of truth for this bootstrap step, not just this one time.
- **A second, separate bug this first run also caught**: `subscribeToCalendar` itself initially failed for all 68 calendars with `403 ACCESS_TOKEN_SCOPE_INSUFFICIENT` even though `events.list` on the same calendars succeeded fine. `calendarList.insert` is a write against the calendarList resource, which the `calendar.readonly` scope (used everywhere else in this client) doesn't cover — confirmed via the actual Google error body (`reason: "insufficientPermissions"`, `ACCESS_TOKEN_SCOPE_INSUFFICIENT` on `calendar.v3.CalendarList.Insert`), not guessed. Fixed by adding the narrow `https://www.googleapis.com/auth/calendar.calendarlist` scope alongside `calendar.readonly` (`CALENDAR_SCOPE` in `calendar.ts`) — deliberately not the broad read-write `https://www.googleapis.com/auth/calendar` scope, which would over-grant event/calendar-data write access this sync never needs; `calendar.calendarlist` only affects the service account's own list membership.
- Verified end-to-end with real data after both fixes: `discovered: 68` (matches the Apps Script's own count), `autoMatched: 1` (the one school that already existed in `schools` at the time — `Edison Park K-8 Center`), `issuesRaised: 67` — expected, not a matching bug: the other 67 real schools simply don't have `schools` rows yet (Phase 2's Lists tab hasn't been populated with the real ~68-school roster). Populating `schools` with the real roster is the next step before most of these auto-match; until then they sit correctly in the "Calendars needing attention" queue.

### Two more real bugs, found only after the real ~72-school roster was imported and the sync run for real

Both surfaced from an actual production run, not code review — confirmed with direct evidence (RPC calls, timestamp reconstruction) before being called bugs, not just suspected.

**1. Stale/orphaned `calendar_sync_issues` rows when a calendar later succeeds.** A calendar flagged as an issue in one run (e.g. before its matching school existed) that later auto-matches in a subsequent run never had its old issue row resolved — nothing in the auto-match branch touched `calendar_sync_issues` at all. Caught by reconciling counts: 49 pinned schools + 67 "open" issues = 116, for only 68 real calendars; the union of distinct `calendar_id`s across both sets was exactly 68, proving 48 of the 67 "open" issues were for calendars that were, in fact, already correctly pinned — just never marked resolved. Fixed in `syncAllCalendars`'s auto-match branch: after successfully pinning a school, it now also resolves any open issue row for that same `calendar_id`. The 48 pre-existing stale rows were resolved manually once as a one-time cleanup; the fix prevents recurrence.

**2. Two calendars sharing a name could silently steal each other's pin.** Google Calendar allows duplicate calendar names (confirmed real: "South Dade Senior High" and "Dr. William Chapman Elementary" each exist as two separate calendars with different ids, per the Apps Script's own share log). `classifyDiscoveredCalendar` only checked whether *this calendar* was already pinned (`pinnedCalendarIds`), never whether the *target school* was already pinned to a *different* calendar. Both same-named calendars independently score a perfect match against the one real school; whichever is processed second in a later run would silently overwrite the first's link via a plain `UPDATE`, with no error (no unique-constraint violation, since it's an update to the same row, not a duplicate insert). Confirmed live: calling `match_school_calendar("South Dade Senior High")` directly returned a perfect (1.0) match, while the second calendar's *stale* issue row (from bug #1, never refreshed) still showed a bogus near-zero score — proving the theft was imminent on the next run, not yet materialized. Fixed by adding a second guard set, `pinnedSchoolIds` (by school id, not calendar id), threaded through `classifyDiscoveredCalendar` (new optional 4th param) and updated on every auto-match within the same pass; a calendar whose best match is a school already claimed by a different calendar is now flagged `school_already_linked` instead of auto-matching. Both duplicate-named pairs, plus one near-miss ("Treasure Island Elementary" naturally fuzzy-matching best against the already-claimed "Redland Elementary"), correctly landed in the review queue after the fix, confirmed against live data.

Net result after both fixes, verified against the real roster: 49 schools correctly pinned, exactly 19 genuinely open issues (9 no match, 7 ambiguous, 3 school-already-linked) — `49 + 19 = 68`, matching the real calendar count with no double-counting.

## Implementation-level decisions (made during Phase 4 build)

### The open session IS the "Demand" — no separate demands table (user-confirmed)
The brief framed the feedback obligation as a blocking "Demand" and asked to confirm how it's modeled. The user's decision: **do not build a demands table.** An `attendance_sessions` row with `clock_out_at IS NULL` (an *open* session) is itself the obligation. Consequences, all falling out of that one rule:
- **Feedback and clock-out are one atomic action** (`clock_out_with_feedback` RPC): it's the only thing that sets `clock_out_at`, so you can never clock out without feedback, and `clock_out_at` is stamped at feedback-submission time.
- **"Can't clock in until the form is filled"** = `clock_in()` refuses when the caller already has an open session. Enforced twice: a friendly `raise` in the RPC, and a **partial unique index** `attendance_one_open_session_per_teacher on (teacher_id) where clock_out_at is null` as a race-proof backstop.
- **"Can't exit the form"** = the `FeedbackForm` has no cancel/close control; the only resolution is submission.
- **"Pops up on login"** = every clock surface re-derives the open session server-side, so it reappears on the next render regardless of logout. No client state to lose.

There is deliberately no "clock-out initiated" intermediate state and no way to record a clock-out time separately from feedback — the two were unified per the decision, which is simpler and impossible to get into a half-clocked-out state.

### Gate scope: block clock-in, don't hijack navigation (user-confirmed)
Chosen over a global layout redirect. A teacher with an open session sees the feedback form on `/clocking` (the clock-in flow is not offered) and on the dedicated `/feedback` route, plus a prominent re-prompt card on the home dashboard — but Schedules/Reports/Settings stay reachable. The **authoritative** block is server-side in `clock_in()`; the UI reads (`getOpenSession`) are for display only and deliberately fail *open* on a transient read error, because the RPC still rejects the clock-in. Defense in depth, UI convenience layered on a server gate.

### Geofence is checked server-side in the RPC; the TS haversine only drives live UI
`clock_in()` re-derives the distance from the school's stored `lat/lng` with the existing `haversine_meters()` SQL function and rejects if it exceeds `schools.geofence_radius_m` — the client's lat/lng are an input to that check, never trusted as the verdict. `src/lib/geo/haversine.ts` (unchanged) powers only the live "you're N m away / move closer" UI, exactly as NEXT_STEPS anticipated. Low GPS accuracy (worse than 100 m, can't confirm presence inside a 200 m fence) is its own client error state with a retry, alongside permission-denied / GPS-off / timeout.

### On-time/late lives in one constant, mirrored in SQL and TS
`ON_TIME_GRACE_MINUTES = 5` in `src/lib/attendance/status.ts` is the single source of truth for the default window; `clock_in(p_grace_minutes integer default 5)` mirrors it. "Configurable" (per the brief) means changing those two defaults, not editing inline literals — a per-school/global override UI is deferred. Late := clock-in strictly more than the grace past the scheduled start; early or unscheduled := on-time. The TS twin (`computeClockInStatus`) is used client-side only to *preview* the status; the RPC computes and stores the authoritative value (`clock_in_status`), kept auditable against later calendar edits via a `scheduled_start_at` snapshot on the session.

### `client_key` idempotency column, even though Phase 4 is online-only
`attendance_sessions.client_key uuid unique` + `clock_in(p_client_key)` dedupe a double-tapped Clock-In (and set up the plan's offline-replay idempotency for a later phase) — a same-key call returns the existing session instead of inserting a duplicate. Cheap now, and it closes a real double-submit gap even online.

### `geo-map.tsx` reuses the `/public/leaflet` string-URL marker pattern
Same reason as `lists/leaflet-map.tsx` (a static PNG `import` throws "iconUrl not set" under this Turbopack version). The map draws the school as the default pin, the teacher as a coloured `CircleMarker` with a GPS-accuracy halo, and the geofence as a `Circle` coloured green/red by whether the fix is inside. Loaded via `next/dynamic({ ssr:false })` from the clocking client component (Leaflet touches `window` at import).

### Migration numbered `0008`, not `0005` as the brief said
The Phase 4 brief (written before earlier phases finished) said `0005_attendance.sql`; `0005`–`0007` are long taken (schools / events / calendar-sync-issues). Used `0008_attendance.sql`, consistent with NEXT_STEPS. Same renumbering call as every prior phase.

### Zoho export deferred; a `zoho_synced_at` column reserves the seam
The product decision is to push each feedback submission to Zoho *after the fact* via API. That exporter is a later phase (needs the `ZOHO_*` creds already stubbed in `.env.local`). Phase 4 stores the feedback in-app and leaves a nullable `zoho_synced_at` marker for that future job — no Zoho code shipped now.

## Implementation-level decisions (Phase 4 hosted verification, later session)

### Fixed: 0007's column-level `REVOKE` never actually worked, on any environment
Confirmed live (`information_schema.column_privileges`) that `authenticated` could still raw-`UPDATE`/`INSERT` the four calendar-match columns on `schools` after `0007_calendar_sync_issues.sql` had been applied — and `tests/calendar-sync-rls.test.ts`'s "cannot bypass with a raw UPDATE" case was failing as a result. Root cause: Postgres's privilege model checks column-level grants **only when no table-level grant already covers the operation** for that role; `0005_schools.sql` grants `insert, update` at the table level (`grant select, insert, update on table public.schools to authenticated`), so `0007`'s `revoke update (col1, col2, ...) ... from authenticated` was a no-op from the moment it was written — this was never a deployment/environment problem, it's a bug in the 0007 migration itself that a purely-textual review missed and only running the actual RLS test against a live grant caught. `DECISIONS.md`'s original reasoning for *why not a trigger* (SECURITY DEFINER's `resolve_calendar_issue()` needs `auth.uid()` to still resolve to the calling manager) remains correct; the *fix* is to revoke the table-wide `insert`/`update` and re-grant them back explicitly per-column, excluding the four protected ones (`supabase/migrations/0009_calendar_column_revoke_fix.sql`). Confirmed fixed: the same test now passes, and a direct SQL check of `information_schema.column_privileges` shows no `authenticated` grant on the four columns. Audited every other migration for the same column-level-`REVOKE`-only pattern (`grep`); `0007` was the only instance — `profiles.role`/`region`/`archived_at` use the (correct) trigger approach instead.

### Hosted verification completed via Supabase MCP, not `supabase db push`

A later session had a connected Supabase MCP server (`apply_migration`, `execute_sql`, `list_migrations`, `get_advisors`) against the same project (`vgyogyojxlvhiwujidhy`), which was used instead: `list_migrations` showed history only through `0006` (confirming `0007` was applied out-of-band, consistent with the REVOKE bug above), then `0008` and `0009` were applied via `apply_migration`. ## Implementation-level decisions (Phase 4 feedback rework — Zoho form + webhook, replacing the in-app form)

Product change, user-confirmed after Phase 4 shipped: the PRD calls for the class-feedback form to be a **Zoho-hosted form**, embedded in the app, with **Zoho's own webhook** (not the teacher's client) closing the attendance session. This replaces the in-app `FeedbackForm` + `clock_out_with_feedback` RPC built earlier in Phase 4 — see the product-level decision table below (updated) and `supabase/migrations/0010_zoho_feedback_close.sql`.

### `clock_out_with_feedback` dropped; `close_session_from_zoho` (service_role-only) replaces it
The in-app form was the only path that submitted feedback, so gating clock-out on it made sense as a single authenticated RPC. Now the Zoho form is the *only* place feedback is captured, and the webhook that reports a submission has no teacher session — it's a server-to-server call authenticated by a shared secret, not a JWT. So the closing RPC moved from "authenticated, checks `auth.uid()` owns the session" to `close_session_from_zoho()`: **not** `SECURITY DEFINER`, **not** granted to `authenticated`/`anon` at all — only to `service_role`, which the webhook route handler (`src/app/api/zoho-feedback/route.ts`) uses via a dedicated admin client (`src/lib/supabase/admin.ts`). A teacher's own JWT can never reach it (confirmed: `tests/attendance-rls.test.ts` asserts this). It's also idempotent (webhooks retry): closing an already-closed session returns the existing row instead of erroring.

### Embedded iframe + hidden field, not a redirect (user-confirmed)
The teacher never leaves the app; the Zoho form is embedded in `/feedback` and `/clocking`'s feedback-gate section via `<iframe>`. The session id is passed as a prefilled hidden field via a URL query parameter (Zoho's documented — though only community-confirmed, not officially pinned down — mechanism for prefilling an embedded form by field **Link Name**), so the webhook payload can echo it back and correlate to the right `attendance_sessions` row.

### Closing only on webhook confirmation, not on iframe return (user-confirmed)
Considered and rejected: optimistically closing the session as soon as the teacher navigates back from the iframe. That would mean the DB gate has no actual proof feedback was submitted (a teacher could bail out of the Zoho form entirely and still get clocked out). Chosen instead: the session stays open until the webhook lands; the client polls its own row every 4s (`src/app/(app)/feedback/feedback-form.tsx`) since there's no reliable cross-origin "submitted" postMessage contract to rely on from inside Zoho's iframe. Tradeoff accepted: a real but short (seconds, not minutes in practice) delay window between submission and the UI updating.

### Offline: local draft in Dexie, submitted through Zoho once back online (user-confirmed)
A Zoho-hosted iframe fundamentally cannot load or submit without connectivity, so "works offline" (the original in-app form's property) can't carry over literally. Instead: offline, a small native form with the same questions saves to IndexedDB via Dexie (`src/lib/attendance/offline-feedback-db.ts` — the first thing to actually use the Dexie dependency that's been an unused placeholder since Phase 0) keyed by session id. Once connectivity returns, the Zoho iframe renders prefilled with the saved draft's values (same URL-param mechanism as the session id), so the teacher just reviews and hits submit inside Zoho's UI — that final submission is still what triggers the webhook and the real close. The open-session gate (already server-side, unchanged) keeps blocking the next clock-in for the entire offline period regardless, so correctness doesn't depend on the draft mechanism at all — it's purely a UX convenience so answers aren't lost.

### Webhook contract: shared secret header + JSON body, `session_id` field name configurable (user-confirmed, UNVERIFIED against a real Zoho account)
Zoho Forms' own webhook docs (fetched and read, not guessed) don't pin down one fixed payload shape — it depends on the "Content Type" and "Payload Parameters" the operator selects per-webhook in Zoho's UI, and no built-in HMAC signature scheme is documented (a third-party aggregator claimed one exists; Zoho's own help article makes no mention of it, so it isn't relied upon). Given that, the contract is defined on **our** side and the operator configures Zoho to match it: `POST` with `Content Type = application/json`, a `Custom Header` named `x-zoho-feedback-secret` carrying a shared secret (`ZOHO_FEEDBACK_WEBHOOK_SECRET`, compared with a timing-safe check), and a body keyed by the form's field Link Names (`session_id`/`rating`/`summary`/`challenges`/`students_present` by default, each overridable via `ZOHO_FEEDBACK_FIELD_*` env vars to match whatever the operator actually names them in the Zoho form builder). The handler also accepts the body wrapped one level under a `data` key, since some Zoho integration surfaces wrap submission data that way and this couldn't be tested against a real delivery. **No real Zoho form/account exists yet** — the full pipeline (secret check → RPC → idempotent retry) was verified live by simulating a Zoho delivery with `curl` against the real hosted DB (disposable session, cleaned up after); the actual Zoho-side config (field Link Names, prefill URL param support in practice, Content-Type selection) still needs a real form. See NEXT_STEPS.md for the exact setup steps once one exists.

### Two real bugs found while building this, both confirmed live
1. **The auth proxy was redirecting the webhook to `/login`.** `src/proxy.ts`'s matcher ran every path except a short static-asset allowlist through the sign-in check, which caught `/api/zoho-feedback` too — Zoho's server-to-server call has no session cookie, so every delivery got a `307` to `/login` instead of reaching the route handler (confirmed via `curl`, which doesn't follow redirects by default, so the bug initially looked like a false pass in a naive test and only showed up once a real browser or `curl -L` followed the chain to `net::ERR_TOO_MANY_REDIRECTS`-style failure). Fixed by excluding `api` from the proxy matcher entirely — `/api/*` route handlers do their own auth (this one via the shared secret; any future session-based ones via `requireRole` inside the handler itself), matching how static assets are already excluded.
2. **A hydration mismatch from reading `navigator.onLine` at `useState` init time.** `FeedbackForm`'s offline/online branch initialized its state as `typeof navigator === "undefined" || navigator.onLine` — deterministic-looking, but `navigator` doesn't exist during SSR (server always takes the `true` branch) while the client evaluates the real value, which can differ from the server's assumption at the exact instant of hydration. Caught live in the Next.js dev overlay ("Recoverable Error", a `<p>` element present only on the client) after seeding a real open session and rendering `/clocking`. Fixed by always initializing to `true` (matching what SSR renders) and syncing the real value in a `useEffect` (with an explicit `eslint-disable` + comment, since this is a legitimate "read an external-system value once at mount" case despite the lint rule's default nagging about `setState` in effects).

### The form's public share link (`zfrmz.com`) is one fixed URL, prefilled with school/teacher/date/class (user-confirmed) — but it refuses to be iframed (confirmed live, unresolved)
Follow-up after the initial rework shipped: the user clarified the real Zoho form is **one single, fixed URL** — `https://zfrmz.com/MIVJGi5IlokeTf8oTsDR`, the same link already embedded in every calendar event's description under "Teacher Feedback:" — not a per-event/per-class URL as a screenshot initially suggested (that screenshot's structured description, "Class Code"/"Pay Rate"/"Teacher Feedback", turned out to be a red herring for THIS purpose; the sync pipeline stores it verbatim, unparsed, per a live grep — no per-class link exists). The user's actual ask: auto-fill school name, teacher name, date, and class from the app, leaving only rating/summary/challenges/students-present for the teacher to type. Implemented by extending `zoho-feedback.ts`'s URL-builder with four more prefill fields (`school`/`teacher`/`date`/`class` Link Names, same env-var-overridable pattern as before) and threading the teacher's `profiles.full_name` down into `FeedbackForm` (previously only available via `requireRole()`'s return value, which the page components weren't capturing).

**Then a real, confirmed blocker surfaced from actually testing it live** (not from code review): `curl -I` on the real `zfrmz.com` URL returns `X-Frame-Options: SAMEORIGIN`, and loading it in the app's `<iframe>` in a real browser aborts (`net::ERR_ABORTED`, empty embed box) — confirmed against the live app with a disposable session, not simulated. This almost certainly means `zfrmz.com` is Zoho's "share as a standalone link" artifact (meant for direct visits, e.g. from a calendar invite), while Zoho Forms' actual iframe-embedding support (which does exist per their own docs) issues a *different* URL/code from the form's Share → Embed tab. **Unresolved as of this entry** — waiting on whoever has Zoho Forms access to check for that embed-specific URL and swap `ZOHO_FEEDBACK_FORM_URL` to it once found (see NEXT_STEPS.md's "🔴 BLOCKER" callout). ### Env vars added; the old Phase 9 "push feedback to Zoho" plan is now probably obsolete
`ZOHO_FEEDBACK_FORM_URL` / `ZOHO_FEEDBACK_FIELD_*` / `ZOHO_FEEDBACK_WEBHOOK_SECRET` added to `.env.example`/`.env.local` (all blank — no real form yet). Not deleted in this pass — that's a product call for whoever owns Phase 9, not something to infer silently.

## Implementation-level decisions (correcting the feedback schema to match the REAL Zoho form)

### The whole feedback field schema was invented, and wrong — corrected by reading the real form's HTML
0010's `close_session_from_zoho(rating, summary, challenges, students_present)` and the webhook/prefill code built alongside it were designed **without ever looking at the real Zoho form** — a mistake, not a placeholder-pending-confirmation situation like the rest of this rework. The real form:
- `Dropdown` — Teacher Name (choices are real teacher full names, each with an `emailassign_val` attribute tying it to a specific email — interesting, but not something the URL-prefill mechanism can use directly, since prefill selects by the dropdown's `value`, i.e. the display name text, not the hidden email)
- `Date` — a date field whose own `date_format` hint is `dd-MMM-yyyy`
- `Dropdown1` — School (choices are real school names, matching this app's `schools.name` closely)
- `Dropdown2` — "Choose program" (choices are class/program names — "Drumline", "Modern Band", etc., matching `calendar_events.summary` for the classes seen so far)

- `MultipleChoice1` — "Did you encounter any issues during today's class?" (Yes/No)
- `MultipleChoice2` — a conditional follow-up, issue status (4 long descriptive choices), only relevant when `MultipleChoice1` = Yes
- `MultiLine` — free-text notes/comments (instrument needs, repairs), optional

No hidden `session_id` field exists on the real form at all.

Reasoning was the user's own — inventing a numeric mapping bakes in an assumption about ordering/weighting ("No answer due to class cancellation" doesn't fit anywhere on a severity scale) that's a real product/reporting decision, not something to default silently. ### Offline draft form rebuilt to mirror the real choices exactly
The offline fallback form (`src/app/(app)/feedback/feedback-form.tsx`'s `OfflineDraftForm`) previously asked for a 1–5 rating + free-text summary/challenges/students-present — none of which exist on the real form. Rebuilt as: an engagement `<select>` populated from `ENGAGEMENT_OPTIONS` (the exact choice strings read out of the real form, so a saved draft's value is always one of Zoho's own valid options and can be prefilled into the live form later without a mismatch), a Yes/No radio group for "had issue," a conditional issue-status `<select>` (only shown/required when "Yes," populated from `ISSUE_STATUS_OPTIONS`), and an optional notes textarea.

Rather than have the policy join through `attendance_sessions` on every row check, `clock_in()` copies `school_id` onto each `gps_checks` row it creates — same trade-off Phase 4 already made implicitly by storing `school_id` directly on `attendance_sessions` rather than deriving it from the event on every read.

### `notify_recipients_for_school()`: Regional Manager(s) of the school's region, falling back to OM/CPO (user-confirmed, not in the original brief)
The brief only says "notifies the Regional Manager" for the GPS case and doesn't say who to notify if the school's region has no RM assigned yet, or the school itself has no region. Asked directly: fall back to OM/CPO (escalate up, same shape as the calendar-sync issue queue being OM/CPO-visible) rather than silently creating no `notification_queue` row. This single helper is reused by both `record_gps_check()` (gps_out_of_fence) and `detect_late_clockins()` (late_clock_in) so the fallback rule lives in one place.

### `detect_late_clockins()` also queues a manager notification, not just a flag (extrapolated from the brief)
The brief explicitly says "notifies the Regional Manager (write to notification_queue, don't send yet)" for the GPS out-of-fence case, but for late-detection only says "creates a flag and a manager escalation card." Since the escalation card's whole point is prompting a manager to act *now*, a `notification_queue` row was added for late-detection too (same `notify_recipients_for_school()` call) — the escalation card itself renders from `flags`/`/flags`, not from the queue; the queue row is what would eventually trigger a push/email once Phase 7's dispatcher exists.

### A 30-minute lookback bounds `detect_late_clockins()`, and a unique index makes re-flagging impossible
`detect_late_clockins()` only considers classes whose `start_at + 5 min` fell within the last 30 minutes — without a bound, a cron job that had been down for hours (or a first-ever run against old seed data) would mass-flag every stale missed clock-in in the table's history at once. The function's own `not exists` check against `flags` is the primary guard against re-flagging the same missed clock-in on every subsequent cron tick; `flags_late_clock_in_once`, a unique index on `(event_id, teacher_id) where type = 'late_clock_in'`, is a race-proof backstop (same "trigger logic + DB constraint as backstop" pattern Phase 4 used for one-open-session-per-teacher).

### GPS sampling is a silent, always-mounted client component polling on foreground, not a push-driven scheduler
`src/components/gps-check-sampler.tsx` is mounted once in the `(app)` layout (teachers only) rather than embedded per-page. On mount and on `visibilitychange` → visible, it starts a 30 s `setInterval` that checks the caller's own open session for any `gps_checks` row past its `due_at` and still `pending`, and if found, takes one `getCurrentPosition` fix and calls `record_gps_check` for each due check. Locking the phone (`visibilitychange` → hidden) clears the interval; any due-but-unsampled checks simply sit `pending` until `close_out_overdue_gps_checks` marks them `unverifiable` — matching the brief's "best-effort while foregrounded" framing exactly, with no attempt at background/service-worker GPS access (which the geolocation API doesn't support unattended anyway).

### `/flags` lives under the existing `(app)` route group, not a new `(manager)` group (user-confirmed, deviates from the brief's literal path)
The brief said `app/(manager)/flags/*`, but every other manager-only page (`/users`, `/lists`) already lives under the single `(app)` group with `requireRole()` gating inside the page component — there's no precedent for a second route group, and adding one would mean duplicating the shared header/back-button/nav-tile chrome from `src/app/(app)/layout.tsx` for this one route. Built at `src/app/(app)/flags/` instead, added to `ROUTE_ROLES` (`MANAGER_ROLES`) in `src/lib/auth/roles.ts`, and given a nav tile.

### `check-closeout` and `late-detect` Edge Functions mirror `calendar-sync`'s shape exactly; neither is deployed or cron-wired yet
Same shared-secret-header + service-role-client pattern as `supabase/functions/calendar-sync/index.ts` (`CHECK_CLOSEOUT_SECRET`/`x-check-closeout-secret` and `LATE_DETECT_SECRET`/`x-late-detect-secret`, `verify_jwt=false` since pg_cron/pg_net calls carry no user JWT). Each is a thin wrapper calling its one RPC (`close_out_overdue_gps_checks`/`detect_late_clockins`) and returning a count. Deploying + scheduling both is a manual step, same as calendar-sync's still-undeployed 5-minute cron — see NEXT_STEPS.md. Both function directories fall under the existing blanket `supabase/functions` exclusion in `tsconfig.json`/`eslint.config.mjs`, so no config changes were needed there.

### Verification: a dedicated hosted RLS test file, not a one-off manual/curl check
Given the acceptance criteria are mostly server-side (RPC results, RLS visibility, notification rows), `tests/gps-checks-rls.test.ts` was added to `npm run test:rls` (same disposable-user-via-service-role pattern as every other RLS suite) rather than only exercising this by hand in a browser: clock-in seeds exactly 5 checks at the right offsets; an in-fence `record_gps_check` call verifies with no flag; an out-of-fence call raises a flag + RM notification and stays invisible to the teacher's own query; the in-region RM sees it and the other region's RM doesn't; `resolve_flag` enforces the region gate; `close_out_overdue_gps_checks` closes a backdated pending check as `unverifiable` with no flag; `detect_late_clockins` flags a missed clock-in exactly once even across repeated runs. All 7 tests pass against the hosted project (confirmed both standalone and alongside the pre-existing `attendance-rls` suite — the full six-file `test:rls` run intermittently hits Supabase's own auth `signInWithPassword` rate limit when every suite's disposable users sign in back-to-back in one process, a pre-existing environment characteristic of this multi-suite pattern, not something introduced by this migration).

## Implementation-level decisions (made during Phase 6 build)

All three of the crux decisions below were confirmed with the user before implementing (the brief explicitly asked to confirm the idempotency-key design and conflict handling first).

### The offline queue covers clock-ins and GPS samples only — NOT feedback/clock-out (user-confirmed)
The Phase 6 brief lists four queue types: clock-ins, clock-outs, feedback submissions, GPS results. But by the time Phase 6 started, the Phase 4 feedback rework had already removed the only teacher-side session-close path (`clock_out_with_feedback` was dropped in `0010`): a session now closes **exclusively** via `close_session_from_zoho()` (service_role), fired by Zoho's webhook when the teacher submits the Zoho form. "Clock-out" and "feedback submission" are therefore the *same* Zoho-webhook action, and there is deliberately no teacher-authenticated RPC that closes a session. Routing feedback through `/api/sync` would mean either duplicating Zoho's role or re-introducing the exact teacher-side close path the rework intentionally deleted. And feedback already has a working offline story from Phase 4 (a Dexie draft prefilled into the Zoho form on reconnect; the open-session gate keeps blocking the next clock-in the whole time, so correctness never depended on the draft). Presented this and the user chose to scope the queue to the two purely-teacher-authenticated RPCs — **clock-ins and GPS samples** — and leave feedback on its existing path. The "done when" acceptance criteria are entirely about the clock-in path, so this loses no required behaviour.

### `/api/sync` runs as the teacher (cookie JWT), not service_role (user-confirmed)
The replay endpoint uses the SSR cookie-based server client (`getUser()` → 401 if signed out) and routes each item to the same SECURITY DEFINER RPCs (`clock_in`, `record_gps_check_offline`) the online path uses. So authorization is *identical* whether a clock-in happens live or via replay — no second trust path to keep in sync, no signed-teacher-id-in-payload surface. The server still re-validates the geofence and clamps the timestamp inside those RPCs regardless. Chosen over a service_role endpoint that would trust a client-supplied teacher id.

### Idempotency key = one client-generated `client_key` UUID per queued item; exactly-once is enforced server-side, not by the client deleting the queue (user-confirmed)
Every queued mutation carries a `client_key` UUID minted at enqueue time. For clock-ins that key *is* the session's `client_key`, so exactly-once falls straight out of Phase 4's existing `attendance_sessions.client_key unique` constraint + `clock_in()`'s idempotent-replay SELECT: a replayed key returns the existing row instead of inserting a duplicate. For GPS samples, idempotency is the check's own status — `apply_gps_sample()` no-ops on any check that isn't `pending`, so re-resolving the same check is a no-op. The client's own guards (a single-flight lock in `sync.ts`; only deleting a queue row on a confirmed `accepted`) are defence-in-depth on top of the server guarantee, not the guarantee itself — so a *forced* double-replay (the acceptance criterion) is safe even if the client-side lock were bypassed. This is why the "done when" holds: the server, not the client, is what makes replay exactly-once.

### Offline clock-in trusts the client's clock-in time, clamped; the geofence is still server-authoritative (user-confirmed)
For an offline replay, `clock_in()` uses the client-recorded `p_clock_in_at` as the authoritative `clock_in_at` (and the basis for on_time/late), so a teacher who clocked in on-time offline and synced hours later isn't wrongly recorded as late. It's **clamped**: `least(p_clock_in_at, now())` (never accept a future time) and floored at `now() - 24h` (a queue older than a day is almost certainly a bug, and backdating a full day would silently rewrite reporting). The geofence, by contrast, is re-derived server-side against the school's stored coordinates exactly as online — the client's lat/lng are an input, never the verdict — so an offline clock-in that was outside the fence is still rejected on sync (tested). This matches the product-decision-table rule ("Offline clock-in trust: auto-accepted, labeled offline; server re-validates on sync") while fixing the one thing that rule doesn't spell out: *which* timestamp is authoritative.

### `record_gps_check_offline` addresses a check by (session client_key, due offset), because the server-side check id doesn't exist offline
An offline clock-in's five `gps_checks` rows are created by `clock_in()` **server-side**, so in pure airplane mode they don't exist anywhere yet — the offline client has no check id to reference. So the offline GPS RPC takes the session's `client_key` (which the client does have — it minted it) plus the check's due offset in minutes, finds the session, then finds the check whose `due_at = clock_in_at + offset` (with a nearest-pending fallback if the exact `due_at` drifted). `/api/sync` processes clock-ins before GPS samples in the same request, so a sample for a session clocked-in in the same batch resolves against the freshly-created checks. Since an offline clock-in is backdated, its checks' `due_at` are already in the past at sync time — the still-`pending` ones the closeout cron would otherwise mark `unverifiable` are exactly the ones an offline sample resolves first (idempotent either way).

### Factored `record_gps_check`'s resolution into `apply_gps_sample()` rather than duplicating the flag/notify logic
The offline GPS path needed the same out-of-fence flag + `notify_recipients_for_school` fan-out as the online one. Rather than copy that ~20-line block into a second function (drift risk — this codebase's `notify_recipients_for_school` comment explicitly calls out keeping "who gets notified" in one place), extracted it into an internal `apply_gps_sample(check_id, lat, lng, accuracy, sampled_at, origin)` that both `record_gps_check` (online, `now()`) and `record_gps_check_offline` (offline, clamped sample time) delegate to. Callers verify ownership before calling; the helper has no client grant. `record_gps_check`'s external behaviour is unchanged — confirmed by the Phase 5 `gps-checks-rls` suite still passing 7/7 after the refactor.

### Dexie for both the queue and the schedule cache; the SW only wakes clients for background sync
The brief says "service-worker caching of schedule data and school coordinates." Chose to hold the *structured* schedule/coords in a Dexie `scheduleCache` store (written by the clocking page while online) rather than parse them out of a cached RSC payload — the clocking screen's on-device geofence needs typed lat/lng/radius, and reconstructing that from a cached HTML/RSC response is fragile. The service worker still caches the app shell (Serwist `defaultCache` + the `/~offline` fallback, both from Phase 0) so the page itself loads offline. For Background Sync, the SW's `sync` handler doesn't re-implement the IndexedDB drain + authenticated fetch — those live in the page context (the Supabase cookie session and the Dexie queue are both there), so it just `postMessage`s open clients to run `syncNow()`. Consequence (documented in NEXT_STEPS): if no tab is open when connectivity returns, the queue drains on the next visit instead of truly headlessly — acceptable for this app.

### `clock_in()`'s signature changed by dropping the old overload, not adding a parallel one
Adding `p_origin`/`p_clock_in_at` via `create or replace` with a longer arg list would have created a *second* `clock_in` overload (Postgres keys functions by argument types), leaving two ambiguous versions. So the migration `drop function`s the old 6-arg signature first, then creates the 8-arg one; the two new params default, so the online caller (which passes `p_event_id`..`p_client_key` by name via PostgREST) still resolves cleanly. Same "one canonical function, redefined" intent as Phase 5 redefining `clock_in` to seed gps_checks.

### Migration numbered `0013`, hosted version timestamp-stamped
`0013_offline_sync.sql` locally; applied via the Supabase MCP `apply_migration` tool, which stamps a timestamp-based version on the hosted project (`2026072114...`) — same local-sequential-vs-hosted-timestamp split documented for every phase since `0008`. Not a re-run risk; the local name and hosted version key are allowed to differ.

## Debugging finding (post-Phase-6): "clock out does nothing" traced to two separate, unrelated Zoho config gaps — neither is a Phase 6 bug

After Phase 6 shipped, the user tried the real end-to-end feedback flow and reported clock-out silently doing nothing — the session never closed. Root-caused across two follow-up sessions, confirmed each time by checking `attendance_sessions` directly (the same open test session, `f8e52696-2000-41dd-972c-808ac51ffae8`, never changed across both):

1. **First pass (testing on `localhost`)**: `ZOHO_FEEDBACK_WEBHOOK_SECRET` was blank in the developer's `.env.local` (the route hard-rejects any call with `500 "Webhook not configured"` when it's empty — confirmed by simulating the webhook with `curl` and getting exactly that error). Separately and more fundamentally, `localhost` is never reachable from Zoho's servers at all, so even with the secret fixed, a real submission could never land.
2. **Second pass (after deploying to Vercel at `https://ymu-a-navy.vercel.app` and setting the same two env vars there)**: same symptom, same untouched DB row. This time the app's own side was actually ready — the gap was that **nothing had configured the webhook on Zoho Forms' own side** (Integrations → Webhooks tab inside the Zoho form editor). Setting env vars in Vercel only prepares the *receiving* end; Zoho still has to be told, on its own UI, to *call* that URL with that header. The user confirmed they'd only ever touched Vercel, never opened Zoho's Integrations tab, and currently has no access to that Zoho account to fix it — deferred to whoever gets access next (see NEXT_STEPS.md's 🔴 section for the exact steps, including re-checking whether the hidden `session_id` field — flagged as missing since the Phase 4 rework — has been added yet).

Neither issue is a defect in anything built during Phase 4 or Phase 6 — both are the same category of gap called out since the original Phase 4 rework entries above ("full webhook round-trip still unconfirmed with a real Zoho submission"): a webhook has **two** independent sides that must both be configured and kept in sync (URL + shared secret), and this is the first time both were actually attempted end-to-end rather than simulated with `curl`. Worth remembering for Phase 7's own webhook/dispatcher surfaces if any: verify both sides explicitly, don't assume one side being correct implies progress on the other.
